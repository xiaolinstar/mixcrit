# P0.12 简化调酒循环验证 Spec

> 创建日期：2026-06-25  
> 产品阶段：P0.12  
> 目标发布标识：0.0.14 (14+)  
> 状态：发布候选（方向试验版）  
> 前置基线：P0.11 / 0.0.13 调酒主舞台打磨

---

## 1. 迭代定位

P0.12 不再以“继续打磨写实模拟细节”为主要目标，也不急于扩第二杯酒。

P0.12 的核心问题是：

> mixcrit 作为养成类游戏，单杯调酒是否可以从完整真实流程简化为短时高频小玩法，同时保留调酒感和顾客反馈可解释性？

---

## 2. 目标

- 将 Mojito 单杯流程压缩到 20-40 秒的常规订单节奏。
- 减少重复点击，尤其是 15ml 量杯多次装填。
- 引入客人口味偏好，让调酒有轻量决策。
- 保留可学习的调酒知识：用量、风味影响、手法原因必须在选择和反馈中可见。
- 建立新的出杯反馈维度：配方、偏好、手法、呈现、速度。
- 为 P0.13 配方系统化和 P1.0 养成 MVP 确定玩法基线。

---

## 3. 非目标

- 不新增第二杯酒。
- 不做完整经营系统。
- 不做库存、进货、金币经验。
- 不重写所有 SpriteKit 视觉节点。
- 不删除当前真实调酒实现；必要时保留为旧流程或参考实现。

---

## 4. 用户体验目标

玩家进入订单后看到：

1. 顾客点 Mojito，并给出一个偏好，例如“清爽一点”“薄荷重一点”“不要太甜”。
2. 调酒台展示 Mojito 的核心原料和 2-3 个可调项。
3. 玩家看到每个简化选项背后的调酒含义，例如 45ml 白朗姆、15ml 糖浆、8 片薄荷。
4. 玩家完成一个倒酒目标区动作。
5. 玩家选择冰量/薄荷/甜度等简化调节，并理解它们对酒体、甜酸、香气和稀释的影响。
6. 玩家完成一次搅拌或摇酒节奏动作。
7. 出杯后看到顾客反馈、表现标签和简短调酒复盘。

---

## 5. 简化流程草案

### 5.1 标准流程

```text
订单卡：Mojito + 客人口味偏好
→ 原料确认：朗姆 / 青柠 / 薄荷 / 苏打
→ 倒酒动作：按住或滑动到目标区
→ 偏好调节：甜度 / 薄荷 / 冰量三选一或档位选择
→ 搅拌/摇酒：短节奏动作
→ 出杯反馈
```

### 5.2 操作数量目标

| 项目 | 当前 P0.11 | P0.12 目标 |
|------|------------|------------|
| 白朗姆 | 3 次装量杯 + 3 次入杯 | 1 次目标区倒酒 |
| 青柠汁 | 1 次装量杯 + 1 次入杯 | 合并到配方基础项 |
| 糖浆 | 1 次装量杯 + 1 次入杯 | 作为甜度档位 |
| 苏打水 | 3 次点击 | 1 次完成或自动补足 |
| 薄荷 | 4 次点击 | 1 个偏好档位 |
| 冰块 | 6 次点击 | 1 个冰量选择 |
| 摇酒 | 3 次点击 | 1 次节奏动作 |

---

### 5.3 知识表达原则

P0.12 的简化不能只留下“轻 / 标准 / 重”这种抽象档位。玩家每次选择都应该学习一个调酒判断：

| 控制项 | 不推荐表达 | 推荐表达 |
|--------|------------|----------|
| 白朗姆 | 轻 / 标准 / 重 | 35ml / 45ml / 55ml，并说明酒感与平衡 |
| 糖浆 | 轻 / 标准 / 重 | 8ml / 15ml / 24ml，并说明酸甜平衡 |
| 薄荷 | 轻 / 标准 / 重 | 4片 / 8片 / 12片，并说明拍香与苦味风险 |
| 冰块 | 轻 / 标准 / 重 | 3块 / 6块 / 8块，并说明降温和稀释 |

简化的是操作复杂度，不是调酒知识本身。

---

## 6. 客人口味偏好

P0.12 先支持 4 种偏好：

| 偏好 | 玩家应对 | 反馈影响 |
|------|----------|----------|
| 清爽一点 | 少糖、苏打感更强、冰量适中 | 偏好匹配加分 |
| 甜一点 | 糖浆档位提高 | 偏好匹配加分 |
| 薄荷重一点 | 薄荷档位提高 | 香气加分 |
| 酒感强一点 | 朗姆目标区略高 | 酒感加分，但过量扣分 |

---

## 7. 出杯反馈模型草案

系统内部仍然需要计算分数、金币和经验，但玩家侧不应把“系统评分”和“顾客反馈”拆成两套信息。P0.12 统一为 **出杯反馈系统**：

```text
内部评价模型
→ 生成口味/手法/速度等标签
→ 由顾客台词表达满意或不满意
→ 结算金币、经验、好感
```

也就是说：

- **系统负责计算**：维度、权重、收益、解锁条件。
- **顾客负责表达**：一句主反馈 + 2-3 个表现标签。
- **UI 只呈现一套结果**：顾客反应、标签、收益变化。

| 维度 | 权重建议 | 输入 |
|------|----------|------|
| 配方匹配 | 30% | 核心原料是否完整 |
| 偏好匹配 | 30% | 甜度、薄荷、酒感、冰量是否贴合客人 |
| 手法表现 | 20% | 倒酒目标区、搅拌/摇酒节奏 |
| 呈现 | 10% | 装饰、冰量、出杯完成度 |
| 速度 | 10% | 完成时间 |

顾客反馈不要只说“少了 15ml”，而要说：

- “这杯更清爽，适合客人的偏好。”
- “糖浆偏多，薄荷的清香被压住了。”
- “摇拌不足，入口层次有点散。”

## 8. 技术方案

### 8.1 新增模型草案

| 类型 | 作用 |
|------|------|
| `CustomerPreference` | 表达客人口味偏好 |
| `SimplifiedMixState` | 记录简化调酒过程的档位和动作结果 |
| `TasteProfile` | 酒感、酸度、甜度、清爽度、香气 |
| `ServeFeedbackEvaluator` | P0.12 出杯反馈评价器，内部计算分数并生成顾客反馈 |

### 8.2 复用现有模块

| 现有模块 | 复用方式 |
|----------|----------|
| `MixingStationView` | 可先在当前调酒页内切换为简化流程 |
| `MixingScene` | 继续承担杯具、液体、倒入、出杯视觉 |
| `MojitoIngredient` | 作为核心原料定义的来源 |
| `ScoreView` | 后续可改名或重构为出杯反馈页，显示顾客反应、标签和收益 |

### 8.3 风险控制

- 不直接删除旧 `MojitoMix` 和 `JiggerState`。
- 新流程可以用 feature flag 或 Debug 参数进入。
- 先做 Mojito 一杯验证，避免同时承担多酒品抽象风险。

---

## 9. 验收标准

- [x] 新玩家首次完成 Mojito 时间不超过 75 秒。
- [x] 熟练玩家完成一杯常规订单不超过 40 秒。
- [x] 常规流程点击/手势次数少于当前 P0.11 的一半。
- [ ] 至少 4 种客人口味偏好能影响顾客反馈。
- [x] 玩家能从出杯反馈页理解“客人为什么满意或不满意”。
- [x] 玩家能从简化选项中看到真实用量和风味影响。
- [ ] 保留足够调酒感：倒酒、搅拌/摇酒、出杯至少各有一次明确反馈。
- [x] `xcodebuild` Debug 模拟器构建通过。

### 10.1 验收结论

P0.12 最小闭环成立，但第一次验收暴露出“简化过度抽象”的问题。修正方向是：继续降低重复操作，同时把真实用量、风味影响、手法原因放回 UI 和反馈中。下一步应复验玩家是否能在 20-40 秒流程内学到 Mojito 的基础比例和调味逻辑。
