# P0.11 调酒页视觉与动画打磨规划

> 创建日期：2026-05-26
> 更新日期：2026-06-20
> 产品阶段：P0.11（延续，不进 P0.12）
> 发布标识：0.0.13 (13)

> 状态：规划中（待逐项实施）

---

## 1. 背景

P0.11 已完成主舞台构图重排与基础动作反馈，真机验收确认 UI 优于 `0.0.12`。但整体仍停留在「看得懂、构图合理」的程度，距离「精致、可信、动作有重量感」还有差距。

经判断：**不立即扩第二杯酒，继续在 P0.11 / 0.0.13 范围内打磨调酒页视觉与动画**。2026-06-25 产品方向调整后，扩配方顺延到 P0.13，P0.12 先验证简化调酒循环。本文档作为 P0.11 打磨工作的统一 backlog、技术选型与验收依据。

---

## 2. 目标

- 让单杯 Mojito 的调制画面达到「精致、可信、动作有反馈重量」的质感基线。
- 形成可复用到后续多酒品的视觉与动画规则（杯具质感、液体、动作节奏）。
- 在不扩功能的前提下，把核心体验打磨到值得扩大测试与扩配方的程度。

## 4. 是否需要引入动画库？

### 4.1 结论（推荐策略）

| 阶段 | 策略 | 说明 |
|------|------|------|
| **当前（P0.11 前半）** | **不引入第三方动画库** | 继续用 **SpriteKit + SwiftUI 原生动画** 把质感打底 |
| **中期（P0.11 后半）** | **按需、点状引入** | 仅对 1–2 个「难用代码做像」的序列评估 Lottie / Rive |
| **长期（P0.12+）** | 建立资产管线 | 统一「静态 PNG + 程序化液体 + 可选动效 JSON」规范 |

**理由：**

1. 项目当前 **零外部依赖**（见 `AGENTS.md`），主舞台已在 `MixingScene.swift` 用 SpriteKit 实现；问题主要是 **分层、遮罩、粒子、资产** 不足，不是缺动画框架。
2. 调酒核心动画（液面变化、加冰、倒入）需要 **跟游戏状态实时联动**（当前多少 ml、几颗冰）。Lottie/Rive 等预烘焙动画很难跟数值状态无缝绑定，除非做大量变体或状态机，成本反而更高。
3. 引入库会带来：SPM 依赖、包体积、设计师导出流程、运行时内存与维护成本。在还没有参考图、风格未锁定时，过早引库容易 **锁死工作流** 却换不来质感提升。

**什么时候才值得引库？**

- 某段动画 **反复调代码仍不像**（例如：出杯礼花、摇酒壶大幅摆动、复杂水花）。
- 已有 **设计师可稳定产出** 的 Lottie/Rive 源文件，且该动画 **不依赖实时配方数据**。
- 团队愿意维护「动效资产目录 + 版本号 + 导出规范」。

### 4.2 技术栈分工（推荐）

```
┌─────────────────────────────────────────────────────────┐
│  SwiftUI（MixingStationView）                            │
│  · 顶部 HUD、底部原料 Dock、动作按钮                      │
│  · 推荐态/选中态：scale、shadow、symbolEffect             │
│  · 页面转场、引导浮层                                     │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  SpriteKit（MixingScene）—— 主舞台动效主战场              │
│  · 杯具、液体分层、冰块/薄荷 z 序与遮罩                   │
│  · SKAction：晃动、落冰、量杯反馈、出杯抬升               │
│  · SKEmitterNode：倒酒粒子、气泡、加冰溅水（待加）        │
│  · SKCropNode / SKMaskNode：液体裁切进杯壁（待加）       │
└─────────────────────────────────────────────────────────┘
                          │
              可选（后期点状）
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Lottie-ios / RiveRuntime（仅 1–2 个装饰性序列）        │
│  · 出杯完成、摇酒夸张镜头、教学手势等                     │
│  · 不用于液面高度、配方进度等状态驱动内容                 │
└─────────────────────────────────────────────────────────┘
```

### 4.3 第三方方案对比（供后期决策）

| 方案 | 适合 mixcrit 的场景 | 不适合的场景 | 引入成本 | 推荐时机 |
|------|---------------------|--------------|----------|----------|
| **继续 SpriteKit 原生** | 倒入粒子、液面、落冰、摇杯、状态联动 | 极复杂角色动画 | 低（已在用） | **现在** |
| **SwiftUI Animation / PhaseAnimator** | 按钮、HUD、原料卡片微交互 | 杯中液体与裁切 | 低 | **现在** |
| **Lottie (lottie-ios)** | 出杯礼花、装饰性水花、教学手势 | 液面随 ml 变化 | 中：SPM + JSON 资产 | T9 重做仍不满意时 |
| **Rive (rive-ios)** | 可交互/state machine 的装饰动画 | 仍需大量美术资源 | 中高：新工具链 | 有 Rive 设计师时 |
| **Spine** | 2D 骨骼角色（未来酒保/客人） | 当前无角色需求 | 高 | P1 养成有角色后 |
| **Unity / Godot 嵌入** | — | 原型过重 | 极高 | **不推荐** |
| **Metal 自定义 shader** | 液体折射、杯壁高光 | 开发周期长 | 高 | P0.11 后期可选 T2 进阶 |

**默认决策：P0.11 不新增 SPM 依赖；T4/T7/T9 先用 SpriteKit 粒子 + 分层实现，验收后再决定是否引 Lottie。**

## 5. 无参考图时的风格锚点（替代参考图）

在缺少外部参考图时，用 **风格支柱（Style Pillars）** 代替，避免审美漂移：

| 支柱 | 描述 | 可验收的具象标准 |
|------|------|------------------|
| P1 摄影感吧台 | 暖顶光、深色台面、杯具是画面主角 | 空杯截图中杯具占主舞台 ≥ 60% 视觉面积 |
| P2 玻璃可信 | 杯壁有高光与厚度，液体在杯内而非贴图前 | 液体边缘被杯壁裁切，非矩形浮在杯前 |
| P3 动作可读 | 每次点击 0.3s 内能感知「发生了什么」 | 倒入/加冰/摇酒/出杯各有 distinguishable 动效 |
| P4 UI 退后 | HUD 与 Dock 提供信息但不抢戏 | 主舞台与底部 Dock 对比度可控，无大面积炫光 |
| P5 克制 | 不堆粒子、不闪屏、不卡通夸张 | 单次动作粒子数 < 30，持续 < 1.2s |

每完成一轮打磨，用 **before/after 截图 + 真机 10 秒录屏** 对照上述支柱，而不是对照某张参考图。

## 6. 动画分类与实现路线

### 6.1 按「是否状态驱动」分类

| 类型 | 示例 | 实现方式 | 是否适合引库 |
|------|------|----------|--------------|
| **状态驱动** | 液面高度、冰块数量、薄荷片数、量杯液体颜色 | SpriteKit 节点 + 数值插值 | 否 |
| **事件驱动** | 点击加冰落一颗、量杯入杯倒出、摇酒晃动 | SKAction + 短粒子 | 一般否 |
| **仪式驱动** | 出杯完成、首次引导手势 | SKAction 或 Lottie | 可评估引库 |

### 6.2 分动作技术方案（细化）

| 动作 | 目标观感 | 技术方案（优先顺序） | 新增资产需求 |
|------|----------|----------------------|--------------|
| **原料入量杯** | 液体「灌入」量杯、量杯微晃 | 1) 量杯内液面 SKShape 插值上升 2) 环形高光 SKAction 3) 可选 5–8 颗粒子 | 量杯 `jigger_fill_mask` 可选 |
| **量杯入杯** | 弧线倒酒 + 杯内液面上升 | 1) `SKEmitterNode` 细流粒子沿贝塞尔 2) 液面 `scaleY` settle 3) 弱化静态箭头 | 倒酒粒子贴图 32×32 |
| **直倒（苏打/薄荷）** | 瓶口到杯口短流 | 粒子 + 对应 ingredient 颜色 | 同倒酒粒子 |
| **加冰** | 冰从杯口落入、溅起、堆叠 | 1) 每颗冰独立 SKSprite 轨迹 2) 落点 3–5 溅射粒子 3) 静态冰层 z 序堆叠 | 可选 `ice_splash` 粒子 |
| **摇酒** | 杯身椭圆轨迹 + 液面晃荡 | 1) `glassGroup` 椭圆 path 动画 2) `glassLiquidNode` 延迟晃动 3) 冰/薄荷子节点轻微滞后 | 无 |
| **出杯** | 完成感、克制、不廉价 | 1) 杯周暖光脉冲 2) 微抬升 3) 液面高光扫过 4) **仍不满意再试 Lottie** | 可选 `serve_glow` 序列帧 |
| **气泡（苏打）** | 持续细泡上升 | `SKEmitterNode` 低速上升，苏打存在时开启 | 气泡粒子 |

### 6.3 SpriteKit 能力清单（待落地）

| 能力 | 用途 | 对应任务 |
|------|------|----------|
| `SKCropNode` | 液体裁切在杯壁内 | T2, T3 |
| `SKEmitterNode` | 倒酒流、气泡、溅水 | T4, T7 |
| `SKAction.customAction` | 液面高度插值 | T2, T4 |
| `SKShader`（可选） | 液面渐变、简单折射 | T2 进阶 |
| 子节点 `zPosition` 规范 | 冰/薄荷/液体/杯壁前后关系 | T3 |

## 9. 引入动画库的决策门槛（若未来评估）

必须同时满足以下条件才引入 Lottie/Rive：

1. SpriteKit 方案在真机验收 **2 轮** 仍不达标
2. 有明确 **单点场景**（不是全局替换）
3. 有可交付的 **源文件 + 导出规范**
4. 包体积增加 **< 500KB**（该动效相关）
5. 不阻碍液面/配方等 **状态驱动** 逻辑

记录模板：

```markdown
## 引库决策 — [日期]
- 场景：出杯 / 摇酒 / …
- SpriteKit 版本问题：
- 第三方试点结果：
- 决策：引入 / 不引入
- 理由：
```

## 10. 非目标

- 不扩第二杯酒、不进入 P0.13 配方系统化

- 不在无音频资产时强行上完整音效系统
