# mixcrit 项目现状评价与规划

> 评价日期：2026-05-24  
> 当前版本：P0.7 / v0.0.8

---

## 一、综合评价

### 总评分：B+（7.5 / 10）

作为一个验证"手机上调一杯酒是否好玩"的 P0 原型，mixcrit 已经成功完成了核心命题的论证。从 P0.0 的单文件点击加料，到 P0.7 的长按倒酒 + 量杯系统 + 贴图物件化，7 轮迭代的演进路径清晰、节奏合理。

---

## 二、分项评价

### 2.1 代码架构：B

**优点：**
- Models / Logic / Views 三层分离清晰，职责边界明确
- `MojitoScorer` 作为纯函数评分引擎，无副作用，可测试性好
- `GamePhase` 枚举驱动状态流转，简洁可控
- 无外部依赖，零配置成本

**问题：**
- **MixingStationView 过重（423行）**：它同时承担了倒酒 Task 循环、量杯转移、加冰动画、摇酒逻辑和 UI 布局。这是一个"上帝视图"，随着 P1 引入更多酒品和工具会迅速膨胀
- **ContentView 状态散装**：8 个 `@State` 挂在 ContentView 上，全部通过 Binding 下传，任何新增状态都要改两层文件
- **Mojito 硬编码**：Mix 模型、Scorer、Ingredient 全部绑死 Mojito 一款酒，P1 需要重构为泛化配方系统
- **DispatchQueue.main.asyncAfter**：摇酒和冰块动画用 GCD 延迟，不如 SwiftUI 的 `TimelineView` 或 `Task.sleep` 可控

### 2.2 交互体验：A-

**优点：**
- 长按倒酒 + 量杯刻度吸附的操作流，手感已经很像调酒
- 触感反馈（sensoryFeedback）覆盖了关键操作节点
- 量杯→成品杯的转移动画有仪式感
- 冰块落杯 + 液面晃动 + 气泡上升的反馈层次丰富

**问题：**
- 缺少操作引导（首次使用时用户可能不知道要按住倒酒）
- 苏打水"补满"的交互不够直觉
- 没有"撤销"或"回退"操作，只能全部重置

### 2.3 视觉完成度：B+

**优点：**
- 9 张 PNG 透明贴图已全部接入，材料架视觉风格统一
- 杯中反馈（液面、冰块、气泡、薄荷、青柠）层次分明
- 吧台灯光 + 台面层次的氛围感不错

**问题：**
- 贴图分辨率和风格一致性需要在真机上验证
- 缺少背景场景贴图（酒吧环境）
- 成品展示页的视觉冲击力不足
- 没有音效支撑，"看起来好喝"还差一口气

### 2.4 文档完整性：A

**优点：**
- 产品规划（product-plan.md）非常完整：定位、用户、三层架构、MVP循环、动画规划、数据设计一应俱全
- 版本发布计划（release-plan.md）SemVer 规范、P0~P3 路线清晰
- P0 验收清单、发布清单结构化，可直接执行

**问题：**
- 缺少技术架构文档（当前只有 AGENTS.md 的简略说明）
- 没有记录已解决的交互设计决策（如"为什么量杯吸附到 15ml 刻度"这类上下文）

### 2.5 可测试性：C+

**问题：**
- 没有测试 target，零测试用例
- `MojitoScorer` 是纯函数，非常适合单元测试，但还没有写
- MixingStationView 的交互逻辑深耦合在 View 中，难以隔离测试
- P0 验收清单全部未执行（需要真机/模拟器）

## 四、下一步规划

### 4.1 P0.7 收口（1~2 天）

这是当前最紧迫的事——P0 原型还没有在任何真机/模拟器上跑过完整验收。

| 事项 | 执行环境 | 说明 |
|------|---------|------|
| Xcode 构建验证 | macOS + Xcode | 打开 .xcodeproj，选模拟器，Cmd+R |
| 执行 P0 验收清单 | 模拟器/真机 | 按 p0-acceptance-checklist.md 逐项过 |
| App Store Connect 创建 App | 浏览器 | 需要 Apple Developer 账号 |
| Archive + Upload TestFlight | Xcode | Product → Archive → Distribute App |

> 这 4 项全部需要在你的 Mac 上手动完成，我无法远程执行。完成后 P0 正式收口。

### 4.2 P1 架构重构（第 1 周）

在扩展到 3 款酒之前，必须先做架构重构。建议拆分为：

**Step 1：泛化配方系统**
- 将 `MojitoIngredient` → `Ingredient`（通用材料模型）
- 将 `MojitoMix` → `CocktailMix`（通用调酒状态）
- 新增 `Recipe` 模型（配方定义：材料列表 + 目标用量 + 工具 + 步骤）
- `MojitoScorer` → `CocktailScorer`（基于 Recipe 的通用评分）

**Step 2：拆分 MixingStationView**
- 提取 `PourEngine`（倒酒 Task 循环逻辑）
- 提取 `JiggerController`（量杯状态与转移逻辑）
- 提取 `AnimationCoordinator`（冰块/摇酒/液面动画控制）
- MixingStationView 只做布局和组合

**Step 3：状态管理收敛**
- 新增 `GameState` 或 `GameStore`（ObservableObject / @Observable）
- ContentView 只持有 GameState，不再散装 8 个 @State
- 各子视图通过 GameState 获取/更新数据

**Step 4：测试基建**
- 新增 mixcritTests target
- 为 CocktailScorer 编写单元测试（覆盖标准配方 / 偏差 / 缺料场景）
- 为 PourEngine 编写单元测试

## 五、风险提示

1. **P0.7 尚未构建验证**：这是最大的风险。代码在 Linux 侧无法编译，任何 Swift 语法错误、Asset 引用问题都还没暴露。建议你尽早打开 Xcode 跑一遍。

2. **P1 架构重构范围大**：从 Mojito 硬编码到泛化配方系统，改动量接近重写。建议重构期间 freeze 功能开发，避免边改边加。

3. **贴图资产成本**：每款新酒需要 3~5 张新贴图（瓶身、装饰、成品），10 款酒就是 30~50 张。建议提前规划批次，用 AI 生图 + 人工调整。

4. **音效缺失**：调酒游戏的沉浸感很大程度依赖声音（倒酒、冰块碰撞、摇壶）。P1 不做音效会让体验大打折扣，建议提前准备音效素材。
