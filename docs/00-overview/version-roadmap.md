> 更新日期：2026-06-15  
> 当前 TestFlight 基线：0.0.8 (8) / P0.7  
> 当前开发阶段：P0.9  
> 下一发布标识：0.0.10 (10+)

---

## 1. 唯一阶段体系

mixcrit 只保留一套真正的阶段体系：`P0.x / P1.x / P2.x`。

`0.0.x (build)` 不再被称为“阶段”或“另一套版本体系”，它只是 App Store Connect 需要的发布包标识。

这能避免两个体系同时描述产品进度造成混淆：

| 类型 | 示例 | 是否表示产品阶段 | 用途 |
|------|------|------------------|------|
| 产品阶段 | P0.8 | 是 | 规划、Spec、路线图、迭代范围 |
| 发布版本 | 0.0.9 | 否 | TestFlight / App Store 展示的安装包版本 |
| Build | 9 | 否 | App Store Connect 上传序号，每次上传必须递增 |

当前具体对应关系：

| 问题 | 答案 |
|------|------|
| 用户现在在 TestFlight 安装的是哪个包？ | `0.0.8 (8)` |
| 这个包对应哪个产品阶段？ | `P0.7` |
| 当前正在规划/开发哪一轮？ | `P0.9` |
| 下一次计划上传哪个包？ | `0.0.10 (10+)` |

原则：

- `P0.x / P1.x` 是唯一产品阶段。
- `0.0.x` 是 P0 原型期发布版本，不代表另一个产品阶段。
- Build 号必须每次上传递增，即使发布版本不变。
- 以后进入 P1 后，版本号切换到 `0.1.0` 或更高。

## 3. P0.9 当前定位

P0.9 不是新增内容版本，不扩展新酒单，不做经营系统。

P0.9 的任务是在 P0.8 SpriteKit 调酒台基础上继续修正首次体验问题：

- 玩家是否知道下一步该做什么。
- 调酒台视觉层级是否清楚。
- 原料、量杯、成品杯、动作按钮之间的关系是否可理解。
- 小屏和大屏是否都能单屏完成核心调酒。
- 调酒页是否具备足够写实的杯具、台面、入料反馈和原料可见性。

当前已启动的技术方向：

- 保留 SwiftUI 作为页面 UI 层。
- 使用 SpriteKit 承载中间调酒主舞台。
- 继续以真实杯具资产、台面光影、物料反馈提升可理解性。

## 4. 发布标识推进规则

### 4.1 何时提升发布版本

提升 `0.0.x` 的条件：

- 完成一个可描述的用户体验变化。
- 需要对外测试新体验。
- 需要在 TestFlight 中区分上一轮反馈版本。

示例：

- `0.0.8`：首个外部测试基线。
- `0.0.9`：首轮反馈修正版本。
- `0.0.10`：P0 收口候选。

### 4.2 何时只提升 Build

只提升 Build 的条件：

- 同一功能范围内的热修。
- 修复上传、签名、崩溃或资源缺失问题。
- 不需要在用户层面表达为新版本主题。

示例：

- `0.0.9 (9)`：P0.8 首次上传。
- `0.0.9 (10)`：P0.8 同范围修复。
- `0.0.9 (11)`：P0.8 继续修复。

### 4.3 何时进入 P1

进入 P1 的条件：

- P0 中“调一杯 Mojito”已经被外部测试证明可理解、可完成、有趣。
- 调酒台基础架构支持扩展第二款/第三款酒。
- 已确定经营循环的最小 Spec。
- 具备至少一个版本周期的验收记录。

P1 起点建议版本号：`0.1.0`。
