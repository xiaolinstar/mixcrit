# mixcrit 版本发布计划

> 制定日期：2026-05-19
> 当前 TestFlight 基线：0.0.8 (8) / P0.7
> 当前开发阶段：P0.8
> 下一发布标识：0.0.9 (9)

产品阶段、发布版本、Build 号的总控关系见 `docs/00-overview/version-roadmap.md`。P0.8 起所有功能与重构遵循 `docs/00-overview/spec-development-process.md`。

## 1. 版本号规范

### 1.1 格式

采用 **语义化版本 2.0.0**（SemVer）：

```
主版本.次版本.修订号
  MAJOR    MINOR    PATCH
```

| 级别 | 变化条件 |
|------|----------|
| **MAJOR** | 破坏性变更（≥ v1.0.0 后），如核心玩法重构 |
| **MINOR** | 新增功能（向后兼容），如新增鸡尾酒、新增系统 |
| **PATCH** | 问题修复（向后兼容），如 bug 修复、动画优化 |

### 1.2 Pre-1.0 版本规则

v1.0.0 之前为试水阶段，版本号规划：

| 版本范围 | 含义 | 示例 |
|----------|------|------|
| **0.0.x** | 最小可行原型，验证核心玩法 | `0.0.1` P0 |
| **0.1.x ~ 0.9.x** | 迭代开发，逐步增加功能 | `0.2.0` P1 |

### 1.3 产品阶段与发布标识映射

项目只有一套产品阶段：`P0.x / P1.x`。对外发布时，App Store Connect 还需要发布版本和 Build，二者只是发布包标识，不代表另一套产品阶段。

| 概念 | 用途 | 当前规则 | 示例 |
|------|------|----------|------|
| 产品阶段 | 产品规划、需求讨论、Spec、文档标题 | `P0.7`、`P0.8` | `P0.8 外部测试反馈修正` |
| 发布版本 | TestFlight / App Store 展示版本 | `0.0.x` 原型期逐次递增 | `0.0.8`、`0.0.9` |
| Build | App Store Connect 上传序号 | 每次上传必须递增，即使发布版本不变 | `8`、`9` |
| Git commit | 代码版本追踪 | 提交信息描述阶段或发布意图 | `Prepare P0.7 TestFlight candidate` |
| Git tag | 稳定测试版本标记 | 上传 TestFlight 后可打 `testflight/v版本-build号` | `testflight/v0.0.8-b8` |

版本映射原则：

- `P0.7` 对应首个外部 TestFlight 候选版本：`0.0.8 (8)`。
- `P0.8` 对应首轮外部反馈修正版：`0.0.9 (9)`。
- 同一产品阶段如果需要重新上传修复包，发布版本可保持不变，只递增 build，例如 `P0.8 = 0.0.9 (10)`。
- 出现新的产品阶段或需要对外区分体验变化时，再递增发布版本，例如从 `0.0.9` 到 `0.0.10`。
- 进入 P1 MVP 时，版本号切到 `0.1.0` 或 `0.2.0`，不再继续使用 `0.0.x`。

### 1.4 发布后缀

发布给外部测试时使用后缀：

| 后缀 | 含义 | 示例 |
|------|------|------|
| `-alpha` | 早期测试，功能未完成 | `0.2.0-alpha` |
| `-beta` | 功能完成，测试完善 | `0.2.0-beta` |
| `-rc` | 候选发布，接近正式版 | `0.2.0-rc` |

| 产品阶段 | 发布版本 | 含义 | 目标 |

## 4. 版本发布节奏

### 4.1 发布频率

| 阶段 | 发布节奏 |
|------|----------|
| P1 | 每 2 周一个 alpha/beta 版本 |
| P2 | 每 3 周一个功能版本 |
| P3 | 每 4 周一个功能版本 + 热修复按需 |

### 4.2 发布流程

```
代码完成 → 创建 CHANGELOG → 打 Tag → 构建 TestFlight → 测试 → 发布
     ↓           ↓            ↓           ↓           ↓        ↓
  开发分支    更新日志      git tag     上传 App Store  内测   App Store
```

### 4.3 Git Tag 规范

```bash
# 格式
git tag -a v1.0.0 -m "Release v1.0.0: P1 MVP"

# 列出所有标签
git tag -l

# 推送标签
git push origin v1.0.0
```
